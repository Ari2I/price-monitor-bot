# Безопасность

Краткий чек-лист того, что уже сделано в проекте и на что обратить
внимание при развёртывании. Не претендует на исчерпывающий аудит —
это практический минимум для бота такого масштаба.

## Уже реализовано в коде

- **Токен бота не хранится в коде** — только в `.env`, который
  исключён из репозитория (`.gitignore`) и из Docker-образа
  (`.dockerignore`). В образ секреты не попадают, передаются только
  во время запуска контейнера через `env_file`.
- **Изоляция пользователей**: каждый пользователь бота видит и может
  удалять только свои товары (`owner_chat_id` проверяется в каждом
  запросе к БД — см. `database/repository.py`).
- **Контейнер работает не от root** — в `Dockerfile` создан отдельный
  пользователь `appuser`, у которого нет лишних привилегий даже при
  теоретическом взломе процесса Python внутри контейнера.
- **Нет открытых входящих портов** — бот использует long polling
  (сам стучится в Telegram), а не webhook, поэтому файрвол на
  сервере можно держать закрытым для всех входящих соединений, кроме
  SSH.
- **Никаких механизмов обхода защит сайтов** (ротация IP, подмена
  cookie, антибот-байпас) не реализовано — бот отправляет обычные
  запросы с браузерным `User-Agent`, как любой обычный браузер.

## Что нужно сделать при развёртывании (см. DEPLOYMENT.md)

- Вход на сервер только по SSH-ключу, пароль отключён.
- Файрвол (`ufw`) разрешает только SSH.
- Автоматические обновления безопасности системы включены.
- Реальный `BOT_TOKEN` не публикуется нигде, включая скриншоты для
  портфолио — на скриншотах стоит закрашивать токен вручную.

## Если токен всё же утёк

Если `BOT_TOKEN` случайно попал в публичный репозиторий, чат или
скриншот — его нужно немедленно отозвать через
[@BotFather](https://t.me/BotFather) командой `/revoke` для нужного
бота и получить новый. Старый токен после отзыва перестаёт работать
сразу.

## Ограничения этого раздела

Этот проект — демонстрационный/учебный, не проходил профессиональный
аудит безопасности и не рассчитан на обработку чувствительных
персональных данных. Если бот будет дорабатываться для реального
коммерческого заказчика с доступом к платёжным данным или большим
объёмом персональных данных пользователей — потребуется отдельная,
более глубокая проработка (шифрование БД, ротация секретов, логи
аудита и т.д.), которая не входит в объём текущей реализации.
