# Мониторинг цен конкурентов — Telegram-бот

Telegram-бот на Python, который отслеживает цены товаров в
интернет-магазинах, хранит историю цен и присылает отчёты — как по
расписанию, так и по запросу, в виде текстового сообщения и
Excel-файла.

Лучше всего бот работает с обычными интернет-магазинами на типовых
платформах (Tilda, InSales, Bitrix, WooCommerce и похожих) — такие
сайты в большинстве случаев отдают HTML сразу, без обязательной
JS-подгрузки, и часто публикуют структурированные данные о цене для
поисковых систем, которые бот умеет использовать напрямую.

Крупные площадки с промышленной антибот-защитой (Wildberries, Ozon,
Яндекс.Маркет, Lamoda и подобные) **в рамках этого проекта
сознательно не поддерживаются** — не потому что это технически
невозможно в принципе, а потому что надёжная работа с ними требует
обхода активных антибот-систем (капча, детекция headless-браузера и
т.п.), что находится за пределами объёма и этических рамок этого
проекта (см. «Чего в проекте намеренно нет»). Подробнее — в разделе
«Известные сложные сайты» ниже.

## Возможности

- **Универсальный парсер цены** с двумя источниками данных:
  1. CSS-селектор, указанный вручную — самый точный способ.
  2. Автоматическое определение через структурированные данные
     JSON-LD (`schema.org`), которые многие сайты публикуют для
     поисковиков — не требует ручной настройки.

  Если цена не находится обычным HTTP-запросом — бот автоматически
  пробует загрузить страницу через headless-браузер (Playwright) для
  сайтов с динамической JS-подгрузкой.
- **Добавление товара через инлайн-кнопки**, без свободного текста
  там, где можно ошибиться: способ поиска цены выбирается кнопкой, а
  не набором вручную слова "да"/"нет"/"авто". Перед сохранением бот
  сразу проверяет цену на странице и показывает результат — заранее
  видно, сработал выбранный способ или нет, не дожидаясь `/report`.
- **История цен** в базе данных (SQLite по умолчанию, легко
  переключить на PostgreSQL — см. `DATABASE_URL`), с возможностью
  посмотреть изменения во времени командой `/history`.
- **Отчёты по расписанию**: раз в заданный интервал бот сам проверяет
  цены всех товаров каждого пользователя и присылает сводку в
  Telegram (текстом и, опционально, Excel-файлом).
- **Отчёты по команде**: `/report` — текстовая сводка прямо сейчас,
  `/excel` — выгрузка текущего среза цен в файл `.xlsx`.
- Многопользовательский режим: каждый пользователь бота видит и
  отслеживает только свои товары.

## Источник данных

Бот работает с произвольными сайтами, которые укажет пользователь —
это не единый API, а парсер открытой страницы товара (CSS-селектор
или структурированные данные страницы). Никакие механизмы обхода
защит сайтов (ротация IP, подмена cookie, антибот-байпас) не
используются и не реализовывались — бот отправляет обычные запросы с
браузерным заголовком `User-Agent`, как это делает любой обычный
браузер.

## Установка

Требуется Python 3.10 или новее.

```bash
pip install -r requirements.txt
playwright install chromium
```

Команда `playwright install chromium` нужна один раз — она
скачивает headless-браузер, который используется для сайтов с
динамической подгрузкой цены. Без этого шага бот всё равно будет
работать с обычными (статическими) сайтами, но не сможет
обрабатывать JS-сайты.

## Настройка

1. Скопируйте `.env.example` в `.env`:

   ```bash
   cp .env.example .env
   ```

2. Получите токен бота у [@BotFather](https://t.me/BotFather) в
   Telegram и укажите его в `.env` как `BOT_TOKEN`.
3. Если провайдер блокирует доступ к `api.telegram.org` (актуально
   для многих сетей в России) — настройте `PROXY_URL`, см. раздел
   «Блокировка Telegram Bot API в России» ниже.
4. При необходимости настройте интервал автоматической проверки
   (`CHECK_INTERVAL_MINUTES`) и другие параметры — все они описаны
   комментариями прямо в `.env.example`.

## Запуск

```bash
python main.py
```

## Использование в Telegram

| Команда | Что делает |
|---|---|
| `/start` | Приветствие и список команд |
| `/add` | Пошаговое добавление нового товара (с кнопками) |
| `/list` | Список отслеживаемых товаров с последней известной ценой |
| `/remove ID` | Удалить товар из отслеживания |
| `/report` | Проверить цены прямо сейчас и получить текстовую сводку |
| `/excel` | Проверить цены прямо сейчас и получить Excel-файл |
| `/history ID` | История изменения цены конкретного товара |
| `/cancel` | Отменить текущий диалог (например, добавление товара) |

### Как добавить товар (`/add`)

Диалог состоит из трёх шагов, каждый со своей подсказкой прямо в
чате:

1. **Название товара** — только для вашего собственного списка.
2. **Ссылка на страницу товара.**
3. **Способ поиска цены** — выбирается кнопкой, без набора текста:
   - **🤖 Найти цену автоматически** — бот попробует найти цену через
     структурированные данные страницы (JSON-LD), без ручной
     настройки. Работает не на каждом сайте.
   - **✏️ Указать CSS-селектор вручную** — бот попросит прислать
     селектор текстом и подскажет, как его найти через «Просмотреть
     код» (Inspect) в браузере.

После выбора способа бот **сразу же проверяет цену на странице** и
показывает результат:

- Если цена найдена — показывается сама цена и способ, которым она
  найдена, с кнопками «✅ Сохранить», «🔁 Попробовать другой способ»
  и «❌ Отмена». Так сразу видно, действительно ли выбранный способ
  работает, и можно проверить, что это именно правильная цена, а не
  случайно найденное другое число на странице.
- Если цена не найдена — бот сообщает причину и предлагает указать
  CSS-селектор вручную или отменить добавление.

Дополнительно настраивать переключение между обычным HTTP-запросом и
headless-браузером не нужно — бот делает это автоматически: сначала
пробует обычный запрос, и только если цена не найдена — пробует
Playwright.

### Известные сложные сайты (Wildberries, Ozon, Яндекс.Маркет, Lamoda)

Если ссылка на товар ведёт на один из этих сайтов, на шаге выбора
способа бот покажет отдельное предупреждение о том, что официальной
поддержки в этом проекте для них нет. Важно понимать, почему:

- Официальные API этих площадок дают продавцу доступ только к его
  **собственным** товарам — получить через них цену чужого
  (конкурентного) товара нельзя в принципе.
- У этих сайтов есть промышленная антибот-защита — при проверке в
  рамках этого проекта запрос к Lamoda был заблокирован именно как
  подозрительный (bot detection), а не просто по IP. Это означает
  активное противодействие автоматическим запросам, а не только
  часто меняющуюся вёрстку.
- Обход такой защиты (решение капчи, маскировка headless-браузера
  под настоящий, подмена отпечатков и т.п.) в этот проект осознанно
  не входит — см. «Чего в проекте намеренно нет».
- Живая проверка цены сразу при добавлении товара (см. выше)
  показывает это сразу: если бот не может получить цену на таком
  сайте, вы увидите это в момент добавления, а не постфактум.

Если понадобится реальная надёжная работа именно с этими площадками
— это отдельная, гораздо более сложная задача (по сути, обход
промышленной антибот-защиты), требующая отдельного обсуждения и
выходящая за рамки текущего проекта.

## Формат Excel-отчёта

Файл содержит одну таблицу со следующими колонками: `Название`,
`Ссылка`, `Текущая цена`, `Валюта`, `Предыдущая цена`, `Изменение`,
`Изменение, %`, `Последняя проверка`.

## Определение валюты

Валюта определяется автоматически, отдельно для каждого товара, и
не привязана к рублю жёстко:

- **В режиме "авто" (JSON-LD)** — валюта берётся из явного поля
  `priceCurrency` в структурированных данных страницы (например,
  `"RUB"`, `"USD"`) — это надёжный источник, так как сайт указывает
  его сам, в машиночитаемом виде.
- **При ручном CSS-селекторе** — валюта определяется по символу или
  сокращению рядом с найденной ценой в тексте страницы (например,
  `$`, `€`, `руб`, `USD`). Код не переводит найденное обозначение в
  единый стандарт (ISO 4217) намеренно: некоторые символы
  неоднозначны (например, `¥` используется и для юаня, и для иены),
  поэтому пользователю показывается ровно то обозначение, которое
  было на странице, без домысливания.
- **Если валюту определить не удалось** — в отчётах и Excel-файле
  используется `₽` как значение по умолчанию для отображения. Это
  предположение (большинство отслеживаемых магазинов — российские),
  а не подтверждённый факт о конкретном товаре.

## Диагностика без Telegram (check_price.py)

Иногда нужно проверить, находит ли парсер цену на конкретной
странице, без запуска бота и без Telegram — например, чтобы
подобрать CSS-селектор для нового сайта или разобраться в сетевой
проблеме (доступен ли сайт с текущего IP/прокси). Для этого в
проекте есть отдельная консольная утилита:

```bash
# Автоопределение через JSON-LD
python check_price.py https://example.com/product

# С конкретным CSS-селектором
python check_price.py https://example.com/product --selector "span.price"

# Сразу через headless-браузер, без попытки обычного запроса
python check_price.py https://example.com/product --dynamic
```

Результат выводится сразу в терминал — эта утилита никак не связана
с Telegram-ботом, поэтому подходит и для диагностики ситуаций, когда
сам бот тоже недоступен (например, при отладке блокировок сети или
прокси): не нужно ничего специально отключать или переключать, чтобы
увидеть ответ.

## Блокировка Telegram Bot API в России

Начиная с 2026 года провайдеры и системы ТСПУ в России периодически
блокируют доступ именно к `api.telegram.org` — при этом сам хост
может отвечать на `ping`, но HTTPS-соединение на порт 443 зависает
по таймауту. Если бот запускается, но не отвечает в Telegram, а в
логе видна ошибка вида `Cannot connect to host api.telegram.org` —
это тот самый случай, а не ошибка в коде.

**Как проверить:** выполните `curl.exe -v https://api.telegram.org`
в терминале. Если соединение виснет и падает по таймауту при
рабочем `ping` — доступ к API заблокирован на уровне сети.

**Решение — прокси.** В `.env` предусмотрена переменная `PROXY_URL`
для HTTP или SOCKS5-прокси:

```
PROXY_URL=socks5://user:password@1.2.3.4:1080
```

Если переменная задана, бот автоматически подключается к Telegram
через указанный прокси (используется пакет `aiohttp-socks`, уже
включённый в `requirements.txt`). Если переменная пустая — бот
работает через прямое подключение.

Важно: `PROXY_URL` влияет только на подключение к Telegram, а не на
запросы парсера к сайтам магазинов (они используют `requests` и
`Playwright` напрямую, без прокси) — эти два вида трафика в проекте
независимы друг от друга. Это специально учтено в архитектуре: сайты
магазинов (особенно российские) сами могут ограничивать доступ с
VPN/иностранных IP, поэтому смешивать эти два прокси-контура было бы
ошибкой. Подробнее о том, как это влияет на выбор сервера при
развёртывании — в `DEPLOYMENT.md`.

Без рабочего прокси бот физически не сможет подключиться к Telegram,
пока действует блокировка — это ограничение на уровне сети, а не то,
что можно решить одним лишь кодом.

## Устойчивость к временным сбоям и блокировкам

Сеть и сайты магазинов иногда сбоят или временно ограничивают
доступ, поэтому в проекте заложены два независимых механизма — но
важно понимать, что оба про то, как **корректно реагировать** на
проблему, а не про обход защиты сайта:

**1. Повтор запроса при временных ошибках.** Обычный HTTP-запрос
автоматически повторяется (с паузой, увеличивающейся с каждой
попыткой) при таймауте, обрыве соединения или ответах HTTP 429/5xx —
такие ошибки обычно проходят сами по себе через некоторое время.
Количество повторов и пауза настраиваются в `.env`
(`REQUEST_MAX_RETRIES`, `REQUEST_RETRY_BACKOFF_SECONDS`). Ответ HTTP
403 **не повторяется никогда** — это осознанный отказ в доступе, а
не временный сбой, и повторные попытки только зря нагружали бы сайт.

**2. Отслеживание систематических неудач.** Если для конкретного
товара цену не удаётся получить несколько проверок подряд (по
умолчанию — от 3), бот явно предупреждает об этом в отчёте (`/report`
и по расписанию) и в `/list` — вместо того чтобы молча повторять
попытки бесконечно. Счётчик автоматически сбрасывается при первой же
успешной проверке.

Эти механизмы повышают устойчивость к обычным, временным сетевым
проблемам и дают пользователю понятный сигнал о систематической
проблеме — но не пытаются преодолеть осознанную защиту сайта (капчу,
детекцию бота и т.п.), что соответствует общему принципу проекта:
см. «Чего в проекте намеренно нет».

## Известные ограничения

- Точность извлечения цены зависит от того, насколько стабильна
  разметка сайта: указанный вручную CSS-селектор может перестать
  работать после редизайна, а автоопределение через JSON-LD зависит
  от того, публикует ли сайт такие данные вообще.
- Автоматический фолбэк на Playwright запускает полноценный
  headless-браузер, что заметно медленнее обычного HTTP-запроса —
  для большого числа товаров с динамическими сайтами это стоит
  учитывать при выборе интервала проверки.
- Бот не обходит капчи и активные антибот-системы — для сайтов с
  такой защитой потребуется отдельная доработка, которая намеренно
  не входит в объём этого проекта.
- Некоторые сайты (в том числе крупные российские магазины) могут
  ограничивать доступ с IP-адресов известных VPN/дата-центров —
  если сервер, на котором работает бот, использует такой IP,
  парсинг может не работать даже при корректно настроенном
  селекторе. См. раздел про выбор сервера в `DEPLOYMENT.md`.
- Планировщик рассылает отчёт всем пользователям бота, у которых
  есть отслеживаемые товары; часовые пояса пользователей не
  учитываются — время проверки одинаково для всех.
- Поддержка Wildberries, Ozon, Яндекс.Маркета и Lamoda сознательно
  не реализована (см. раздел «Известные сложные сайты» выше) — эти
  площадки не входят в объём проекта, а не являются "экспериментальной,
  но рабочей" функцией.
- Сравнение "было/стало" в отчётах корректно, только если валюта
  товара не менялась между проверками — смена валюты сайтом является
  редким случаем и отдельно не обрабатывается.

## Чего в проекте намеренно нет

Как и в предыдущих проектах портфолио, здесь отсутствуют функции
ротации/смены IP-адреса, подмены cookie, решения капчи и обхода
антибот-защит сторонних сайтов (детекция headless-браузера, подмена
отпечатков и т.п.) — такие механизмы выходят за рамки задачи
учебного и демонстрационного проекта. Именно поэтому Wildberries,
Ozon, Яндекс.Маркет и Lamoda официально не поддерживаются: надёжная
работа с ними потребовала бы именно таких механизмов.
