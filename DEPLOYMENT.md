# Развёртывание на сервере

## Важное уточнение по выбору сервера

Изначально в этом гайде рекомендовался зарубежный VPS — это самый
простой способ обойти блокировку Telegram Bot API в России без
прокси. Но если бот отслеживает цены на **российских** сайтах
(например, Lamoda), это решение конфликтует само с собой: те же
российские магазины часто ограничивают доступ именно с иностранных
IP и известных VPN/дата-центров — то есть сервер, который выглядит
"иностранным" для Telegram, точно так же будет выглядеть
подозрительным для российского интернет-магазина.

**Из-за этого рекомендация меняется на обратную:**

- **Сам сервер — российский VPS.** Запросы парсера к сайтам
  магазинов идут с обычного российского IP и не выглядят подозрительно.
- **Прокси нужен только для подключения бота к Telegram** — не
  локальный VLESS-клиент на домашнем компьютере (он выключается
  вместе с компьютером и не подходит для работы без вашего участия),
  а отдельный, постоянно работающий прокси-сервер (например,
  небольшой зарубежный VPS с Xray/3proxy, поднятый специально под
  эту задачу). В `.env` он указывается через `PROXY_URL` — код бота
  уже поддерживает это, никаких изменений не требуется (см. основной
  README, раздел «Блокировка Telegram Bot API в России»).

Это разделение работает благодаря архитектуре проекта: `PROXY_URL`
оборачивает только сетевую сессию Telegram-бота (`aiogram`) и никак
не влияет на запросы парсера к сайтам магазинов (`requests`,
`Playwright`) — они физически используют разные библиотеки для
похода в сеть.

Дальнейшие шаги гайда (Docker, автозапуск, бэкапы) одинаковы
независимо от того, какой именно сервер выбран — различие только в
географии сервера и в том, откуда берётся прокси для Telegram.

## 1. Выбор и первоначальная настройка сервера

Подойдёт недорогой VPS у российского провайдера (Timeweb, REG.RU,
Selectel и похожие) — конкретных цен не называю, так как не могу
проверить их актуальность прямо сейчас, сверьтесь на сайте перед
покупкой. Требования минимальны: 1 vCPU, 1-2 ГБ RAM достаточно
(Playwright/Chromium — самая прожорливая часть).

После получения доступа по SSH:

```bash
# Обновляем систему
sudo apt update && sudo apt upgrade -y

# Создаём отдельного пользователя вместо работы под root
sudo adduser botadmin
sudo usermod -aG sudo botadmin
```

## 2. Базовая защита сервера (важно сделать до загрузки кода)

**Отключаем вход по паролю, оставляем только SSH-ключ:**

```bash
# На своей машине, если ключа ещё нет:
ssh-keygen -t ed25519

# Копируем публичный ключ на сервер:
ssh-copy-id botadmin@ваш_сервер
```

Затем на сервере в `/etc/ssh/sshd_config` устанавливаем:

```
PasswordAuthentication no
PermitRootLogin no
```

И перезапускаем службу: `sudo systemctl restart sshd`.

**Включаем файрвол** — боту не нужны входящие соединения (он
использует long polling, а не webhook), поэтому открытым можно
оставить только SSH:

```bash
sudo apt install ufw -y
sudo ufw allow OpenSSH
sudo ufw enable
```

**Включаем автоматические обновления безопасности:**

```bash
sudo apt install unattended-upgrades -y
sudo dpkg-reconfigure --priority=low unattended-upgrades
```

## 3. Установка Docker

```bash
curl -fsSL https://get.docker.com | sudo sh
sudo usermod -aG docker $USER
newgrp docker

# Проверка
docker --version
docker compose version
```

## 4. Загрузка проекта и настройка

```bash
# Например, через git (если репозиторий приватный — потребуется
# настроить SSH-ключ для GitHub отдельно) или просто scp:
scp -r ./price_monitor_bot botadmin@ваш_сервер:~/

cd ~/price_monitor_bot
cp .env.example .env
nano .env   # вписать реальный BOT_TOKEN и PROXY_URL
```

На российском VPS в `PROXY_URL` нужно указать адрес отдельного
стабильного прокси-сервера для Telegram (см. пояснение выше) —
именно он решает проблему блокировки `api.telegram.org`, при этом
сам сервер бота остаётся российским для корректной работы парсера.

Создаём каталог для базы данных и выдаём права пользователю
контейнера (см. пояснение в разделе «Права на каталог данных» ниже):

```bash
mkdir -p data
chmod 777 data
```

## 5. Запуск

```bash
docker compose up -d --build
```

Проверка, что бот работает:

```bash
docker compose logs -f
```

Остановка / перезапуск:

```bash
docker compose down
docker compose up -d
```

## 6. Автозапуск после перезагрузки сервера

`restart: unless-stopped` в `docker-compose.yml` уже гарантирует, что
контейнер поднимется сам после сбоя или перезагрузки — но только
если сама служба Docker запускается при загрузке сервера:

```bash
sudo systemctl enable docker
```

После этого перезагрузка сервера (например, после планового
обслуживания у хостера) не потребует ручного вмешательства.

## Права на каталог данных

Пользователь внутри контейнера (`appuser`) работает не от root, и
у него может не быть прав на запись в примонтированный с хоста
каталог `./data`, если тот принадлежит другому пользователю. Самое
простое решение — `chmod 777 data` перед первым запуском, как
показано выше. Это не самый строгий вариант с точки зрения прав
доступа, но приемлемый компромисс для одного контейнера с
единственным назначением: у SQLite-файла внутри `./data` нет доступа
извне, кроме самого сервера.

## Резервное копирование базы данных

История цен хранится в одном файле SQLite (`./data/price_monitor.db`)
— рекомендуется настроить простое ежедневное копирование на случай
сбоя диска сервера:

```bash
# Пример cron-задачи (crontab -e), копирует базу раз в сутки в 3:00
0 3 * * * cp ~/price_monitor_bot/data/price_monitor.db ~/backups/price_monitor_$(date +\%F).db
```

Куда именно складывать копии (отдельный диск, облачное хранилище) —
зависит от требований конкретного заказчика; готового решения для
переноса на внешнее хранилище в проекте нет.

## Что не входит в этот гайд

- Настройка HTTPS/домена — не нужна, так как бот не принимает
  входящие HTTP-запросы (использует long polling).
- Настройка самого прокси-сервера для Telegram (Xray/3proxy на
  отдельном зарубежном VPS) — это отдельная задача системного
  администрирования, не специфичная для этого проекта.
- Мониторинг падений и алерты (например, через Uptime Kuma или
  аналоги) — можно добавить отдельно, если это станет реальным
  требованием заказчика.
- Миграция с SQLite на PostgreSQL при росте нагрузки — код уже
  поддерживает смену `DATABASE_URL`, но сама СУБД и её резервное
  копирование в объём этого гайда не входят.
